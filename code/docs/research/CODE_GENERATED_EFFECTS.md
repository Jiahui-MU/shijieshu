# Code-Generated Effect Layers

This note records which parts of the chapter intro / hero transition should be generated by code instead of delivered as baked video assets.

## Scope

Current focus:

- Hero-style chapter intro stage.
- Scroll-driven dissolve between visual states.
- Paper / tear / edge transitions between chapters.
- Particle, sparkle, light, and motion details that need to stay alive while the user scrolls or moves the pointer.

Out of scope for this note:

- Final chapter copy.
- Product card contents.
- Exact replacement artwork for each section.

## Asset Roles

Use image or video assets for the actual scene content:

- Room background.
- Emotion world background.
- Character / object foreground, ideally transparent PNG or WebP.
- Portal core if a rendered animation already exists.
- Screen recordings inside UI mockups.

Generate the motion framework in code:

- Scroll progress.
- Sticky stage timing.
- Dissolve mask.
- Tear edge.
- Paper surface.
- Sparkle / dust / glints.
- Parallax and pointer response.
- Video scrub synchronization.

The rule is: visual subject matter can be an asset; behavior and transitions should be code.

## Must Be Code-Generated

### 1. Scroll Progress Controller

Purpose:

- Convert scroll position into a stable `progress` value from `0` to `1`.
- Keep the chapter intro pinned while progress advances.
- Feed the same progress value into all layer transforms.

Why code:

- This is the backbone of the reference interaction.
- It must respond to trackpad/mouse movement, scroll direction, viewport height, and mobile layout.

Expected API:

```ts
type StageProgress = {
  progress: number;
  velocity: number;
  direction: 1 | -1;
  isPinned: boolean;
};
```

### 2. Scroll-Scrub Video Controller

Purpose:

- Map scroll progress to `video.currentTime`.
- Let a supplied video behave like a scroll-controlled image sequence.

Why code:

- Autoplay video will not match the reference.
- The user needs to be able to slow down, reverse, and pause the transition with the trackpad.

Used for:

- `/Users/mujiahui/Desktop/首图.mp4`
- Future screen demos or portal clips.

Expected controls:

- `startProgress`
- `endProgress`
- `easing`
- `holdFirstFrame`
- `holdLastFrame`

### 3. Dissolve Mask

Purpose:

- Reveal one visual layer while another disappears.
- Create a soft, uneven, granular dissolve instead of a simple opacity fade.

Why code:

- It needs to stay tied to scroll progress.
- It should work after changing images.
- A baked dissolve video would force us to re-render every time the artwork changes.

Implementation target:

- CSS `mask-image` where possible.
- Canvas-generated noise mask when the shape needs more control.

Controls:

- `threshold`
- `softness`
- `noiseScale`
- `grainAmount`
- `drift`

### 4. Torn Paper Edge

Purpose:

- Draw the moving boundary where one chapter tears into the next.
- Produce the edge, subtle fibers, and shadow without using a fixed PNG strip.

Why code:

- The edge must move and morph with scroll.
- The same system must be reusable across all chapter transitions.
- A static image strip caused the earlier wrong result: a fixed, pasted-on border.

Implementation target:

- Canvas path generated from seeded noise.
- Separate passes for paper fill, edge highlight, shadow, and fine fibers.

Controls:

- `seed`
- `amplitude`
- `frequency`
- `edgeY`
- `edgeOpacity`
- `paperColor`
- `fiberDensity`
- `shadowStrength`

### 5. Paper Surface

Purpose:

- Fill the revealed area below/inside the tear with paper-like tone and subtle texture.

Why code:

- The paper surface must follow the same tear geometry.
- It should scale cleanly across desktop and mobile.
- It should not require exporting a full-screen paper texture for every viewport.

Implementation target:

- Canvas fill with procedural grain.
- Optional low-opacity paper texture overlay later.

Controls:

- `baseColor`
- `grainOpacity`
- `warmth`
- `vignette`

### 6. Sparkle / Dust / Light Particles

Purpose:

- Add living ambient details: sparkles, dust, glints, small drifting particles, and dissolve flecks.

Why code:

- This is the best candidate for code generation instead of `sparkle-overlay.webm`.
- It can react to scroll and pointer movement.
- It stays reusable when the hero image changes.

Implementation target:

- Canvas 2D for the first version.
- Three.js only if depth or 3D camera movement becomes necessary.

Controls:

- `density`
- `speed`
- `twinkle`
- `colorPalette`
- `depth`
- `scrollInfluence`
- `pointerInfluence`

### 7. Portal Glow / Energy Field

Purpose:

- Support the transition from room to emotion world with a growing/shrinking glow, ring, or field.

Why code:

- The glow needs to align with the scroll state and dissolve mask.
- The supplied stills already show the portal states, but the in-between behavior should be adjustable.

Implementation target:

- Radial gradients and canvas particles.
- Optional video layer for a detailed portal core.

Controls:

- `centerX`
- `centerY`
- `radius`
- `ringWidth`
- `color`
- `intensity`
- `distortion`

### 8. Parallax / Pointer Response

Purpose:

- Make background, foreground, particles, and light layers move subtly with pointer or trackpad input.

Why code:

- The reference page remains alive even after the major transition settles.
- This cannot be baked into one static image.

Controls:

- `pointerX`
- `pointerY`
- `scrollVelocity`
- `layerDepth`
- `settleSpring`

### 9. Section Transition Orchestrator

Purpose:

- Reuse the same intro-stage logic across chapters.
- Decide when a chapter is in intro mode, content mode, or transition-out mode.

Why code:

- The site has multiple chapter boundaries.
- Each transition should use the same timing model, not hand-tuned one-off strips.

Expected model:

```ts
type ChapterIntroTimeline = {
  holdStart: [number, number];
  dissolve: [number, number];
  tear: [number, number];
  settle: [number, number];
  contentEnter: [number, number];
};
```

## Can Be Asset-Based

These do not need to be generated by code:

- Final background illustrations.
- Character illustration frames.
- Yarn ball / product foreground images.
- A rendered portal core, if we want a very specific look.
- Mac or UI screen recordings.
- Static paper texture, if used only as a subtle overlay.
